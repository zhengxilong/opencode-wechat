# 架构说明

[English](./architecture.md) | [简体中文](./architecture.zh-CN.md)

本文档说明 `opencode-wechat` 预期采用的高层架构。

## 1. 总体概览

`opencode-wechat` 的定位是一个连接微信与 OpenCode 的轻量桥接器。

它的目标不是替代 OpenCode 本身，而是提供一个基于消息驱动的访问通道，让用户可以通过微信来操作 OpenCode。

从高层看，这个 Bridge 主要承担四类职责：

- 接收微信消息
- 解析用户意图与项目上下文
- 调用 OpenCode Session
- 将结果回传到微信

## 2. 高层处理流程

核心运行流程如下：

1. 一条消息从微信进入 Bridge。
2. Bridge 对消息进行校验和解析。
3. Bridge 确定应由哪个项目和 Session 处理该消息。
4. Bridge 创建或复用一个 OpenCode Session。
5. Bridge 将 Prompt 发送给 OpenCode。
6. Bridge 处理 OpenCode 返回结果。
7. Bridge 把结果回复到微信。

## 3. 主要组件

### 3.1 CLI 层

CLI 层提供运维入口，例如：

- `start`
- `login`
- `logout`
- `status`
- `doctor`

这一层主要负责启动、诊断和本地运维流程。

### 3.2 配置层

配置层负责加载和校验桥接配置，包括：

- OpenCode 调用参数
- 微信运行时配置
- 项目目录定义
- 访问与安全设置
- 可选的指标配置

### 3.3 微信接入层

这一层负责与微信消息通道集成。

典型职责包括：

- 二维码登录处理
- 登录态复用
- 消息轮询或接收
- 消息发送

### 3.4 解析与路由层

这一层负责解释收到的消息，并决定：

- 这是命令还是普通 Prompt
- 应该路由到哪个项目
- 是否属于已有对话
- 是否需要进入审批流程

### 3.5 Session 层

这一层负责与 OpenCode Session 建立桥接。

它的职责包括：

- 创建 Session
- 复用 Session
- 将 Session 绑定到项目目录
- 发送 Prompt
- 收集 OpenCode 返回结果

### 3.6 状态存储层

状态层用于在本地持久化 Bridge 的运行数据。

典型数据包括：

- 登录状态
- Session 映射关系
- 用户与项目上下文
- 审批状态
- 运行时元数据

在早期版本中，基于文件的持久化通常是最简单也最实用的默认实现。

### 3.7 审批与安全层

这一层负责判断某个动作是否必须在执行前要求用户显式确认。

这是必要的，因为消息通道会让高风险操作看起来过于轻量。

### 3.8 指标与可观测性层

这一层虽然不是强制，但对于以下目标很有帮助：

- 观察 Bridge 健康状态
- 调试和故障分析
- 后续与管理平台做更深集成

## 4. 参考数据流

```text
微信消息
    ↓
微信接入层
    ↓
解析 / 路由
    ↓
项目与 Session 解析
    ↓
审批检查
    ↓
OpenCode Session 调用
    ↓
结果格式化
    ↓
微信回复
```

## 5. 项目绑定模型

Bridge 预期支持一个或多个项目目录配置。

每条消息应当基于以下上下文进行项目解析：

- 显式指定的项目
- 默认项目
- 当前会话上一次已激活的项目

这样可以减少执行歧义，也能让整体行为更容易理解和排查。

## 6. 设计目标

这套架构的主要设计目标包括：

- 保持 Bridge 与 OpenCode Core 解耦
- 让个人开发者部署足够简单
- 让运行状态保持本地、清晰、可理解
- 明确安全与风险边界
- 为未来独立二进制打包留出空间
- 为后续多通道扩展预留 adapter 边界

## 7. 早期版本的非目标

在早期公开版本中，这个项目不打算解决所有问题。

当前非目标包括：

- 完整的企业 IAM 和策略系统
- 大规模多租户编排能力
- 通用聊天机器人平台
- 在 Bridge 内直接内建复杂管理后台能力

## 8. 后续演进方向

后续架构可以沿着这些方向继续演进：

- 更好的测试覆盖和接口契约
- 更强的访问控制
- 更清晰的多通道适配边界
- 独立可分发的二进制打包
- 更深入的指标与审计支持
